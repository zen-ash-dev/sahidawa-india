# PR #1634 — [FEATURE] Add Date Range and Status Filters to CSV Scan History Export

> **Merged:** 2026-06-11 | **Author:** @hrx01-dev | **Area:** Frontend | **Impact Score:** 13 | **Closes:** #1552

## What Changed

We have introduced a new `ExportModal` component to the scan history page, enabling users to export their scan history to CSV with advanced filtering capabilities. This modal allows filtering by specific date ranges (last 7 days, last 30 days, or all time) and by scan status (all, verified, suspicious, or fake). The generated CSV file now includes a dynamic filename that reflects the applied filters, such as `sahidawa_history_fake_30d.csv`.

## The Problem Being Solved

Prior to this change, our scan history page offered only a basic "Export to CSV" function that would export the entirety of a user's scan history without any options for filtering. This limitation made it challenging for users to analyze or share specific subsets of their data, such as only their "fake" scan results or scans performed within a particular timeframe. The generic filename `sahidawa_scan_history.csv` also provided no immediate context about the exported data's content, leading to potential confusion when managing multiple exports.

## Implementation Details

The core of this feature is the new `ExportModal` component, located at `apps/web/app/[locale]/history/ExportModal.tsx`. This component is a React functional component that manages its own state for `dateRange` (initialized to "all") and `statusFilter` (initialized to "all"). It receives `isOpen`, `onClose`, the full `history` array (typed as `ScanHistoryEntry[]`), and a `t` (translation) function as props.

A `useRef` hook, `modalRef`, is attached to the modal's main `div` to facilitate dismissal. We leverage our existing `useOnClickOutside` hook (imported from `@/hooks/useOnClickOutside`) to automatically close the modal when a user clicks outside its boundaries or presses the `Escape` key, enhancing accessibility and user experience.

The `handleExport` function within `ExportModal.tsx` orchestrates the filtering and CSV generation process:
1.  It defines a `rangeMs` object to map string-based date ranges ("7d", "30d", "all") to their corresponding millisecond values, with "all" mapping to `Infinity`.
2.  It filters the `history` prop based on the current `dateRange` and `statusFilter` states. An entry matches the date range if `now - entry.timestamp <= rangeMs[dateRange]`. An entry matches the status if `statusFilter === "all"` or `entry.status?.toUpperCase() === statusFilter.toUpperCase()`.
3.  It constructs the CSV content:
    *   `headers` are defined as `["ID", "Date", "Medicine Name", "Status"]`.
    *   `rows` are generated by mapping the `filtered` history entries. Crucially, `e.medicineName` is wrapped in double quotes (`"${e.medicineName}"`) to ensure that medicine names containing commas are correctly parsed as a single field by spreadsheet applications.
    *   `csvContent` is then assembled by joining headers and rows with commas and newlines.
4.  Finally, it triggers the download: a `Blob` is created with `csvContent` and `type: "text/csv;charset=utf-8;"`. A temporary URL is created using `URL.createObjectURL()`, and an `<a>` element is programmatically created. The `href` is set to this URL, and the `download` attribute is set to a dynamically generated filename (e.g., `sahidawa_history_fake_30d.csv`) using the selected `statusFilter` and `dateRange`. The link is then clicked, and the object URL is revoked to free up resources. After the download, `onClose()` is called to dismiss the modal.

On the `apps/web/app/[locale]/history/page.tsx`, the previous direct `exportToCSV` function has been removed. Instead, we now import `useTranslations` from `next-intl` and the new `ExportModal` component. A `useState` variable, `isExportModalOpen`, controls the visibility of the modal. The "Export to CSV" button's `onClick` handler now simply sets `isExportModalOpen` to `true`. The `ExportModal` component is conditionally rendered at the bottom of the page, passing the necessary `isOpen`, `onClose`, `history`, and `t` props. The type of the `history` state in `HistoryPage` was also updated from `any[]` to `ScanHistoryEntry[]` for improved type safety.

New translation keys for the modal's title, labels, date range options, status options, and button text have been added under the `"ScanHistory"` namespace in `apps/web/messages/en.json` to support internationalization.

## Technical Decisions

1.  **Dedicated `ExportModal` Component:** We opted to create a separate `ExportModal.tsx` component instead of embedding the logic directly into `page.tsx`. This decision promotes modularity, separation of concerns, and reusability. It keeps the main history page component focused on displaying history data, while the export logic and UI are encapsulated, making future maintenance and enhancements more manageable.
2.  **Client-Side CSV Generation:** The CSV generation and download process is entirely handled on the client-side using `Blob` and `URL.createObjectURL`. This approach avoids the need for server-side processing for simple data exports, reducing server load, improving responsiveness, and simplifying the overall architecture for this feature.
3.  **Robust Modal Dismissal with `useOnClickOutside`:** The choice to use our existing `useOnClickOutside` hook for modal dismissal ensures a consistent and accessible user experience. Users can intuitively close the modal by clicking outside it or pressing the `Escape` key, which is a standard UI pattern. This also handles focus management, returning focus to the trigger button for keyboard users.
4.  **CSV Field Quoting:** The decision to wrap `e.medicineName` in double quotes (`"${e.medicineName}"`) during CSV generation is a critical detail. This prevents issues where medicine names containing commas (e.g., "Medicine A, B, and C") would be incorrectly parsed into multiple columns by spreadsheet software, ensuring data integrity in the export.
5.  **Dynamic Filename Generation:** Creating a dynamic filename like `sahidawa_history_fake_30d.csv` based on the applied filters significantly improves the usability of the export feature. It provides immediate context to the user about the content of the downloaded file, reducing the need to open and inspect files to understand their contents.
6.  **`next-intl` for Internationalization:** Integrating `useTranslations` from `next-intl` for all user-facing text within the modal ensures that this new feature is fully internationalized from its inception, aligning with SahiDawa's commitment to supporting multiple languages.

## How To Re-Implement (Contributor Reference)

To re-implement this feature from scratch, a contributor would follow these steps:

1.  **Create the `ExportModal` Component:**
    *   Create a new file: `apps/web/app/[locale]/history/ExportModal.tsx`.
    *   Import necessary React hooks (`useRef`, `useState`), the custom `useOnClickOutside` hook from `@/hooks/useOnClickOutside`, the `ScanHistoryEntry` type from `@/lib/db/scanHistory`, and icons (`Download`, `X`) from `lucide-react`.
    *   Define an `ExportModalProps` interface to type the component's props: `isOpen: boolean`, `onClose: () => void`, `history: ScanHistoryEntry[]`, and `t: (key: string) => string`.
    *   Implement the `ExportModal` functional component:
        *   Initialize `dateRange` and `statusFilter` states using `useState` with default values of "all".
        *   Create a `modalRef` using `useRef<HTMLDivElement>(null)` and attach it to the main modal container `div`.
        *   Call `useOnClickOutside(modalRef, onClose, isOpen)` to enable outside-click and Escape key dismissal.
        *   Add a conditional return `if (!isOpen) return null;` to prevent rendering when the modal is closed.
        *   Implement the `handleExport` asynchronous function:
            *   Define a `rangeMs` object mapping date range strings ("7d", "30d", "all") to their millisecond equivalents.
            *   Filter the `history` array based on the `dateRange` and `statusFilter` states. The filtering logic should compare `entry.timestamp` against `now - rangeMs[dateRange]` and `entry.status?.toUpperCase()` against `statusFilter.toUpperCase()`.
            *   Define `headers` as `["ID", "Date", "Medicine Name", "Status"]`.
            *   Map the `filtered` history entries to `rows`, ensuring that `e.medicineName` is wrapped in double quotes: `` `"${e.medicineName}"` ``.
            *   Construct the `csvContent` by joining `headers` and mapped `rows` with commas and newlines.
            *   Create a `Blob` from `csvContent` with `type: "text/csv;charset=utf-8;"`.
            *   Generate a temporary URL using `URL.createObjectURL(blob)`.
            *   Create a new `<a>` HTML element.
            *   Set `link.setAttribute("href", url);`.
            *   Set `link.setAttribute("download", filename);`, where `filename` is dynamically constructed like `` `sahidawa_history_${statusFilter.toLowerCase()}_${dateRange}.csv` ``.
            *   Programmatically trigger the download: `link.click();`.
            *   Revoke the object URL: `URL.revokeObjectURL(url);`.
            *   Call `onClose();` to close the modal after export.
        *   Render the modal's JSX structure, including a fixed overlay, a centered modal content `div` (attached to `modalRef`), a header with title and close button, and a body containing two `select` elements for date range and status, and a "Generate CSV" button that calls `handleExport`.

2.  **Integrate into `HistoryPage`:**
    *   Modify `apps/web/app/[locale]/history/page.tsx`.
    *   Import `useTranslations` from `next-intl` and `ExportModal` from `./ExportModal`.
    *   Inside the `HistoryPage` component, initialize `const t = useTranslations("ScanHistory");`.
    *   Add a state variable `const [isExportModalOpen, setIsExportModalOpen] = useState(false);` to control the modal's visibility.
    *   Define helper functions `const openExportModal = () => setIsExportModalOpen(true);` and `const closeExportModal = () => setIsExportModalOpen(false);`.
    *   Locate the existing "Export to CSV" button and change its `onClick` handler to `openExportModal`.
    *   At the end of the `HistoryPage`'s return JSX, render the `ExportModal` component conditionally:
        ```tsx
        <ExportModal
            isOpen={isExportModalOpen}
            onClose={closeExportModal}
            history={history}
            t={t}
        />
        ```
    *   Update the `useState<any[]>([]);` for `history` to `useState<ScanHistoryEntry[]>([]);` for better type inference and safety.

3.  **Add Translations:**
    *   Open `apps/web/messages/en.json`.
    *   Add a new top-level key `"ScanHistory"` and populate it with translation keys for the modal title (`export_modal_title`), labels (`export_range_label`, `export_status_label`), date range options (`range_7d`, `range_30d`, `range_all`), status options (`status_all`, `status_verified`, `status_suspicious`, `status_fake`), and button text (`export_button`, `cancel`).

## Impact on System Architecture

This change primarily impacts the frontend user experience and client-side data handling.
*   **Enhanced User Experience:** We have significantly improved the utility of the scan history page by providing users with granular control over their data exports. This allows for more targeted analysis and management of personal health information.
*   **Modular Frontend Design:** The introduction of `ExportModal.tsx` as a standalone component reinforces our commitment to a modular and maintainable frontend architecture. This separation of concerns makes the codebase easier to understand, test, and extend in the future without affecting other parts of the `HistoryPage`.
*   **No Backend Impact:** The entire feature is implemented client-side, meaning there is no direct impact on our backend services, database schema, or API endpoints. This keeps the system's core robust and unaffected by frontend UI enhancements.
*   **Accessibility Improvements:** The integration of the `useOnClickOutside` hook and the implicit focus management it provides contributes to a more accessible user interface, ensuring that keyboard users can effectively interact with the modal.
*   **Foundation for Future Features:** This pattern of using modals for advanced filtering and export functionalities can be replicated for other data-heavy sections of the platform, unlocking possibilities for similar enhancements elsewhere.

    *   The `ExportModal` correctly opens when the "Export to CSV" button is clicked.

    *   **Status Filters:**
        *   "All Statuses": All entries, regardless of status, are included.
        *   "Verified Only": Only entries with `status: "VERIFIED"` are included.
        *   "Suspicious Only": Only entries with `status: "SUSPICIOUS"` are included.
        *   "Fake Only": Only entries with `status: "FAKE"` are included.
    *   **Combined Filters:** All combinations of date range and status filters were tested to ensure accurate data subset generation.
*   **CSV Content and Format:**
    *   The generated CSV includes the correct headers: "ID", "Date", "Medicine Name", "Status".
    *   Each row accurately reflects the filtered `ScanHistoryEntry` data.
    *   Medicine names containing commas (e.g., "Aspirin, 500mg") are correctly wrapped in double quotes within the CSV to prevent parsing errors.
    *   The `Date` column correctly displays the `timestamp` in `toISOString()` format.
*   **Filename Generation:**
    *   The downloaded CSV file's name dynamically reflects the selected filters (e.g., `sahidawa_history_fake_30d.csv`, `sahidawa_history_all_7d.csv`).
*   **Edge Cases:**
    *   Exporting when the history is empty (the "Export to CSV" button is hidden in this scenario, preventing an empty export).
    *   Exporting history with only one type of status or within a very narrow date range.
